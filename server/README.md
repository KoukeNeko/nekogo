# Nekogo Dictionary TTS API

The Go service uses an asynchronous cache-through path:

```text
App ID -> ready audio? -> 200 AAC
                    \-> same kioku-content.db -> background Irodori generation -> AAC + SQLite index
```

The first request for a missing `vocab:<id>` or `example:<id>` queues one deduplicated background generation and immediately returns `404 audio_not_found`, allowing the App to use device TTS without waiting. Later requests return the generated file. The manifest `prewarm` command remains available to avoid first-use misses.

## Storage

- SQLite (`DATABASE_PATH`) maps each stable entry ID and voice profile to an audio object.
- `CONTENT_DATABASE_PATH` points to a read-only copy of the exact `kioku-content.db` version bundled in the App, so the Go server can resolve IDs to canonical Japanese text.
- Audio files use readable entry paths: `audio/vocab/<id>.m4a` and `audio/example/<id>.m4a`. SQLite stores that relative object key, ETag, size, canonical text, versions, and generation status.
- Rows become `ready` only after an audio file has been written successfully.

Persist the entire `./data` directory in production. Do not check it into Git.

## Configure

Go 1.25 or newer is required.

```sh
cd server
cp .env.example .env
cp tts-overrides.example.json tts-overrides.json
set -a
. ./.env
set +a
```

The default `IRODORI_API_MODE=gradio` uses the HTTP API already exposed by the Irodori Gradio app and invokes `ffmpeg` to convert its complete WAV output to AAC/Opus without removing speech segments. Set the mode to `openai` only when the separate OpenAI-compatible Irodori-TTS-Server is running. The default quality profile is linear, 60 steps, text CFG 3.0, speaker CFG 5.0, and deterministic per-entry seeds.

## Export a manifest

From the App project root, export canonical content IDs and Japanese spelling:

```sh
npm run tts:manifest -- --kind vocab --limit 100 --output server/vocab-top-100.jsonl
npm run tts:manifest -- --kind example --limit 100 --output server/example-100.jsonl
```

`--limit` is applied per selected kind; omit it to export every row. `--kind all` exports both vocab and examples.

## Pre-generate audio

```sh
cd server
go run . prewarm -manifest vocab-top-100.jsonl
```

The command skips already-ready rows with the same canonical text and profile. Pass `-force` to deliberately regenerate them. Pronunciation overrides are applied before synthesis when `TTS_OVERRIDES_FILE` is configured.

If audio was generated by another operator tool, import it without calling Irodori. Add a `file` field (relative to the manifest) and run:

```json
{"entry_id":"vocab:2820690","text":"辞書","file":"audio/2820690.m4a"}
```

```sh
go run . import -manifest import-audio.jsonl
```

Imported files must already match the manifest `format` (`m4a` by default; `aac` and `opus` are also supported) and the configured voice/profile.

## Serve cached and on-demand audio

```sh
go run . serve
```

Health check:

```sh
curl http://localhost:8090/healthz
```

Retrieve a ready word or example:

```sh
curl http://localhost:8090/api/v1/dictionary-audio/vocab/2820690 --output speech.m4a
curl http://localhost:8090/api/v1/dictionary-audio/example/42 --output example.m4a
```

Successful responses contain raw audio, `ETag`, `X-Audio-Profile`, `X-Cache: HIT`, and immutable private cache headers. The first known-ID miss contains `X-Audio-Generation: started`; duplicate requests while it is running contain `in-progress`:

```json
{"error":{"code":"audio_not_found","message":"Audio is being generated. Retry later."}}
```

An ID absent from the content DB returns `404 entry_not_found` and does not call Irodori.

## Security

- For the initial trusted LAN build, leave `APP_API_KEY` empty and do not expose port 8090 to the internet.
- Keep `IRODORI_API_KEY` only on this server.
- A production mobile client needs HTTPS and real session authentication; a fixed token stored in an App setting is not a secret.
- Keep the Irodori and Gradio endpoints private. The App calls only this GET-only cache-through API.
