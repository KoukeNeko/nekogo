# Nekogo Dictionary TTS API

The Go service uses an asynchronous cache-through path backed by a persistent
SQLite queue:

```text
App ID / prewarm manifest -> synthesis_jobs
                              |         |
                         GPU worker  CPU worker
                              \         /
                         WAV -> Opus -> ready asset
```

The first request for a missing `vocab:<id>` or `example:<id>` queues one
deduplicated background generation and immediately returns `404
audio_not_found`, allowing the App to use device TTS without waiting. Later
requests return the generated file. Queue leases, retries, and generation tokens
survive process restarts and prevent stale workers from publishing over a newer
regeneration.

## Storage

- SQLite (`DATABASE_PATH`) maps each stable entry ID and voice profile to an audio object.
- `CONTENT_DATABASE_PATH` points to a read-only copy of the exact `kioku-content.db` version bundled in the App, so the Go server can resolve IDs to canonical Japanese text.
- Audio files use readable entry paths: `audio/vocab/<id>.opus` and `audio/example/<id>.opus`. SQLite stores that relative object key, SHA-256 ETag, size, canonical text, versions, and generation status.
- Rows become `ready` only after an audio file has been written successfully.

Persist the entire `./data` directory in production. Do not check it into Git.

## Configure

Go 1.25 or newer is required.

```sh
cd server
cp .env.example .env
cp tts-overrides.example.json tts-overrides.json
set -a
. ./.env
set +a
```

The default `IRODORI_API_MODE=gradio` uses two instances of the Irodori Gradio
API. GPU defaults to `http://192.168.50.169:7860` with `cuda/fp32`; optional CPU
defaults to `http://192.168.50.169:7862` with `cpu/fp32`. Each backend has one
worker. CPU claims only text at or below `CPU_MAX_TEXT_RUNES` and prefers short
vocabulary after honoring interactive priority. Disable CPU with
`IRODORI_CPU_ENABLED=false`; queued work remains available to GPU.

Both workers invoke `ffmpeg` to convert the complete WAV output without removing
speech segments. The output is mono Opus at 32 kbps VBR, `application=voip`, 60
ms frames, and compression level 10. Set the mode to `openai` only when the
separate OpenAI-compatible Irodori-TTS-Server is running. The synthesis profile
is linear, 60 steps, text CFG 3.0, speaker CFG 5.0, and deterministic per-entry
seeds. `IRODORI_BASE_URL` remains a temporary fallback alias for the GPU URL.

## Export a manifest

From the App project root, export canonical content IDs and Japanese spelling:

```sh
npm run tts:manifest -- --kind vocab --limit 100 --output server/vocab-top-100.jsonl
npm run tts:manifest -- --kind example --limit 100 --output server/example-100.jsonl
```

`--limit` is applied per selected kind; omit it to export every row. `--kind
all` exports both vocab and examples. Use `--order learning` for a full manifest
whose numeric priorities interleave useful examples with high-frequency/JLPT
vocabulary:

```sh
npm run tts:manifest -- --kind all --order learning --output server/all-audio-prioritized.jsonl
```

## Pre-generate audio

```sh
cd server
go run . prewarm -manifest vocab-top-100.jsonl
```

The command validates the manifest and durably enqueues missing rows; the
long-running `serve` process owns all GPU/CPU synthesis. It skips already-ready
rows with the same canonical text and profile and can be stopped safely after
enqueue completes. Pass `-force` to rotate the generation token and deliberately
regenerate ready rows. Pronunciation overrides are applied before enqueue when
`TTS_OVERRIDES_FILE` is configured.

If audio was generated by another operator tool, import it without calling Irodori. Add a `file` field (relative to the manifest) and run:

```json
{"entry_id":"vocab:2820690","text":"辞書","file":"audio/2820690.m4a"}
```

```sh
go run . import -manifest import-audio.jsonl
```

Imported files must already match the manifest `format` (`opus` by default; `m4a` and `aac` are also supported) and the configured voice/profile.

## Serve cached and on-demand audio

```sh
go run . serve
```

Run a read-only terminal progress view in another shell:

```sh
go run . status
```

On a terminal, the command updates one progress-bar line with ready/expected,
queued/running/failed counts, GPU/CPU completions, aggregate throughput, and
ETA. It exits automatically at 100%; press Ctrl-C to stop monitoring without
affecting generation. For scripts, `go run . status -once` emits one stable
timestamped line. `-interval 5s` changes the refresh interval.

Health check:

```sh
curl http://localhost:8090/healthz
```

Lightweight queue status (unlike the manifest, this does not scan audio files):

```sh
curl http://localhost:8090/api/v1/dictionary-audio/status
```

It reports profile, ready/expected, queue states, total bytes, and cumulative
GPU/CPU scheduler completions.

Retrieve a ready word or example:

```sh
curl http://localhost:8090/api/v1/dictionary-audio/vocab/2820690 --output speech.opus
curl http://localhost:8090/api/v1/dictionary-audio/example/42 --output example.opus
```

Successful responses contain raw audio, `ETag`, `X-Audio-Profile`, `X-Cache: HIT`, and immutable private cache headers. The first known-ID miss contains `X-Audio-Generation: started`; duplicate requests while it is running contain `in-progress`:

```json
{"error":{"code":"audio_not_found","message":"Audio is being generated. Retry later."}}
```

An ID absent from the content DB returns `404 entry_not_found` and does not call Irodori.

Stream all currently ready assets without a bundle:

```sh
curl http://localhost:8090/api/v1/dictionary-audio/manifest.ndjson
```

The first line contains the schema/profile/format plus expected and ready counts and total bytes. Every following line contains one stable `entry_id`, SHA-256, and byte length, sorted by `entry_id`. Missing or size-mismatched files are excluded and their stale index rows are invalidated. Ready downloads and the manifest are not throttled; only generation misses and deletes use `RATE_LIMIT_PER_MINUTE`.

Delete one generated asset and its DB index. The operation is idempotent; the
next `GET` queues a fresh generation:

```sh
curl -X DELETE http://localhost:8090/api/v1/dictionary-audio/example/42
curl http://localhost:8090/api/v1/dictionary-audio/example/42
```

## Security

- For the initial trusted LAN build, leave `APP_API_KEY` empty and do not expose port 8090 to the internet.
- Keep `IRODORI_API_KEY` only on this server.
- A production mobile client needs HTTPS and real session authentication; a fixed token stored in an App setting is not a secret.
- Keep the Irodori and Gradio endpoints private. The App calls only this scoped dictionary-audio API.
